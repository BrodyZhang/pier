## 2. Core Concept

The website's domain `ailaopo.online` translates to "love your wife". The core message is: if you love your wife/girlfriend, create a dedicated AI Agent for her -- a personalized webpage generated by AI that expresses your love.

## 3. Architecture

```

Web Application Container (Docker)
    ├── Web Server (Nginx/Gunicorn/Werkzeug)

```

## 4. Functional Requirements

### 4.1 Public Pages

| # | Feature | Description |
|---|---------|-------------|
| FR-1 | **Homepage** | Simple, clean landing page introducing the product. Explains the concept: "Love your partner? Create a dedicated AI Agent for them." Must include a CTA to register/login. |
| FR-2 | **Register** | User registers with email only. A verification code is sent to the email. |
| FR-3 | **Login** | User logs in with email + verification code. |
| FR-4 | **About/FAQ** | (Optional) About page explaining the service. |

### 4.2 Registration System

| # | Feature | Description |
|---|---------|-------------|
| FR-5 | **Email Verification** | User enters email → system sends verification code → user enters code to complete registration |
| FR-6 | **Daily Registration Cap** | Maximum 100 new registrations per day (UTC+8). Once limit is reached, registration is closed until next day. |
| FR-7 | **Rate Limiting** | Cooldown for resending verification codes (e.g. 60 seconds). |

### 4.3 User Dashboard (Logged-in)

| # | Feature | Description |
|---|---------|-------------|
| FR-8 | **Dashboard** | After login, user sees a dashboard with their existing AI Agents and the option to create a new one. |
| FR-9 | **Create Agent** | Form with fields: **Agent Name** (required), **Agent Description** (required, free text describing the agent's personality, purpose, tone, etc.). |
| FR-10 | **Agent List** | List all agents created by this user, showing status (pending review / approved / rejected). |
| FR-11 | **Share Management** | For each approved agent, show the shared partner's info and the unique agent URL. |

### 4.4 AI Agent Review System

| # | Feature | Description |
|---|---------|-------------|
| FR-12 | **Review Statuses** | `pending_review` → `approved` / `rejected` |
| FR-13 | **Admin Review Page** | A protected admin page listing all agents awaiting review. |
| FR-14 | **AI-Assisted Review** | When an admin opens an agent for review, the AI first analyzes the description and provides a review suggestion (e.g. content safety check, quality assessment). |
| FR-15 | **Human Must Approve** | The final approval/rejection decision MUST be made by a human. AI only assists, never decides. |
| FR-16 | **Rejection Reason** | If rejected, admin provides a reason visible to the creator. |
| FR-17 | **Review Notification** | After review, the creator receives a notification (in-app and/or email). |

### 4.5 Agent Generation

| # | Feature | Description |
|---|---------|-------------|
| FR-18 | **AI-Generated Page** | Once approved, the AI generates a static HTML page based on the agent name + description. The page should be romantic, personalized, and beautiful. |
| FR-19 | **Agent URL** | Each agent gets a unique URL: `https://ailaopo.online/agent/<unique-id>` |
| FR-20 | **Regenerate** | User can edit the name/description and request regeneration. This creates a new version (see FR-22). |

### 4.6 Access Control

| # | Feature | Description |
|---|---------|-------------|
| FR-21 | **Two-Person Limit** | Each agent page can be viewed by at most 2 people: the **creator** and **one partner**. The creator can "share" the agent by entering their partner's email. The partner must also register/login. |
| FR-22 | **Agent Visibility** | The agent page is only accessible to logged-in users who are either the creator or the shared partner. Unauthorized users see a 403 or login prompt. |
| FR-23 | **Shared View** | Both the creator and the partner see the SAME agent page after login. |
| FR-24 | **Cancel Share** | Creator can revoke sharing. The partner loses access. |

### 4.7 Version Management

| # | Feature | Description |
|---|---------|-------------|
| FR-25 | **Version History** | Every time the agent is regenerated, the previous version is saved. |
| FR-26 | **Rollback** | User can view all versions and rollback to any previous version. |
| FR-27 | **Version Display** | Version metadata: version number, created date, regenerated date. |

## 5. Non-Functional Requirements

| # | Requirement | Description |
|---|-------------|-------------|
| NFR-1 | **Performance** | Page load time < 2s for agent pages. |
| NFR-2 | **Scalability** | System should handle up to 100 registrations/day and ~1000 active agent pages. |
| NFR-3 | **Security** | All traffic via HTTPS. Session tokens secure. Verification codes expire after 10 minutes. |
| NFR-4 | **Mobile-Friendly** | All pages responsive. Works on mobile browsers. |
| NFR-5 | **Database Backups** | Daily automated backup of database. |
| NFR-6 | **Email Service** | Reliable email delivery for verification codes and notifications. (SMTP / SendGrid / etc.) |

## 6. Data Model (Preliminary)

```
User:
  - id (PK)
  - email (unique)
  - created_at
  - last_login_at

Agent:
  - id (PK)
  - user_id (FK → User)
  - name
  - description
  - status (pending_review | approved | rejected)
  - rejection_reason (nullable)
  - unique_slug (UUID)
  - created_at
  - updated_at

AgentVersion:
  - id (PK)
  - agent_id (FK → Agent)
  - version_number
  - name
  - description
  - generated_html (text)
  - ai_prompt_used (text)
  - created_at

AgentShare:
  - id (PK)
  - agent_id (FK → Agent, unique)
  - partner_email
  - partner_user_id (FK → User, nullable)
  - shared_at
```

**Access Rule**: An agent page at `/agent/<slug>` is accessible if:
- The viewer is logged in AND
- (viewer.id == agent.user_id) OR (viewer.id == agent_share.partner_user_id)

**Two-Person Enforcement**: If the creator tries to share with a second partner, the system should either reject or automatically replace the previous share (to be decided).

## 7. Review Flow

```
User submits agent (name + description)
    ↓
Agent status: pending_review
    ↓
Admin sees in review queue → clicks to review
    ↓
AI analyzes description → shows review suggestion
    ↓
Admin reviews AI suggestion + original description
    ↓
Admin approves OR rejects (with reason)
    ↓
If approved → AI generates agent page → version saved → URL created
    ↓
Creator notified → can now share with partner
```

## 11. Constraints

- The current Docker deployment uses `--restart=always` with Let's Encrypt certs mounted from host
- VPS resets daily at 03:00 UTC+8
- Only ports 80 and 443 are open on the firewall
- Existing `.github/workflows/deploy.yml` deploys to a single container named `pier`
- Any new services (backend, database) will need to be added to this deployment setup
- The current `docker-compose.yml` is outdated and unused -- if multi-container setup is needed, we should update it
